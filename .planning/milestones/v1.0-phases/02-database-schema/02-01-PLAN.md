**4. handle_new_user trigger function (SECURITY DEFINER):**
```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER SET search_path = ''
AS $$
BEGIN
    INSERT INTO public.profiles (id, email, full_name)
    VALUES (
        NEW.id,
        NEW.email,
        NEW.raw_user_meta_data ->> 'full_name'
    );
    RETURN NEW;
END;
$$;

CREATE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

**5. organization_members table:**
- `id UUID PRIMARY KEY DEFAULT uuid_generate_v4()`
- `organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE`
- `user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE`
- `role TEXT NOT NULL DEFAULT 'viewer' CHECK (role IN ('admin', 'operator', 'viewer'))` (CHECK constraint, not ENUM — per research)
- `is_active BOOLEAN NOT NULL DEFAULT true` (not soft delete — per research recommendation)
- `joined_at TIMESTAMPTZ DEFAULT NOW()`
- `created_at TIMESTAMPTZ DEFAULT NOW()`
- `updated_at TIMESTAMPTZ DEFAULT NOW()`
- `UNIQUE(organization_id, user_id)`
- Enable RLS
- SELECT policy: `user_id = (SELECT auth.uid())` — direct column check, NO self-referencing subquery (prevents infinite recursion per research pitfall #1)
- No INSERT/UPDATE/DELETE policies (service_role only)
- updated_at trigger
- Indexes: `organization_id`, `user_id`

**IMPORTANT PATTERNS TO FOLLOW:**
- All `auth.uid()` calls MUST be wrapped as `(SELECT auth.uid())` for RLS performance caching
- All policies specify `TO authenticated` role
- Use CHECK constraints for role values (not Postgres ENUM)
- `deleted_at IS NULL` in SELECT policies for tables with soft delete
- SECURITY DEFINER with `SET search_path = ''` on the trigger function
