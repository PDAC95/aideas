Step A. Create `web/src/lib/admin/catalog-queries.ts`.

Export this type and function:

```typescript
export interface AdminCatalogTemplate {
  id: string;
  slug: string;
  displayName: string;
  category: string;
  industry_tags: string[] | null;
  pricing_tier: string;
  setup_price: number | null;
  monthly_price: number | null;
  is_active: boolean;
  is_featured: boolean;
  has_active_automations: boolean;
  active_automations_count: number;
}

export async function fetchAdminCatalogTemplates(locale: string): Promise<AdminCatalogTemplate[]>;
```

Implementation rules:
- Use `createAdminServerClient()` so admin RLS policies (Phase 17) apply.
- Call `assertPlatformStaff` first; on failure throw — the page-level layout already redirects non-staff, so reaching this query without staff is a hard error.
- SELECT from `automation_templates` WITHOUT filtering by `is_active` so inactive templates appear too.
- Embed the resource: `translations:automation_template_translations!left(field, value, locale)` and filter `.eq('translations.locale', locale).eq('translations.field', 'name')`. Use a LEFT join (`!left`) so a template without a translation row still appears (defensive — `displayName` falls back to slug).
- Issue a second query for active automation counts: `select('template_id, status').in('status', ['active','in_setup','paused','pending_review']).is('deleted_at', null)`. Reduce client-side into a `Map<templateId, count>`.
- Sort the merged result by `sort_order ASC, slug ASC`.
- Return the merged array.

Step B. Create `web/src/lib/actions/admin-catalog.ts`.

Two server actions, each with `'use server'` at the top of the file:

```typescript
type ToggleResult = { ok: true } | { ok: false; error: 'invalid_input' | 'not_authenticated' | 'not_staff' | 'update_failed' };

export async function toggleTemplateActive(input: { templateId: string; nextActive: boolean }): Promise<ToggleResult>;
export async function toggleTemplateFeatured(input: { templateId: string; nextFeatured: boolean }): Promise<ToggleResult>;
```

Body:
1. Validate input with an inline Zod schema (`z.object({ templateId: z.string().uuid(), nextActive: z.boolean() })`). On failure return `{ ok: false, error: 'invalid_input' }`.
2. `const sb = await createAdminServerClient();`
3. `const auth = await assertPlatformStaff(sb);` — return `{ ok: false, error: auth.error }` (narrowed to `'not_authenticated' | 'not_staff'`) if not ok.
4. `await sb.from('automation_templates').update({ [field]: value }).eq('id', templateId);`
5. On supabase error (`error != null`), `console.error('[toggleTemplateX] update_failed', { error })` and return `{ ok: false, error: 'update_failed' }`.
6. On success, call `revalidatePath('/admin/catalog')` AND `revalidatePath('/dashboard/catalog')` so the customer side picks up the change on next request.
7. Return `{ ok: true }`.

Do not add deactivation-warning logic in the server action. The warning is a client UX gate. The server action accepts the toggle either direction. If a malicious client bypasses the modal, deactivation still proceeds (per CONTEXT.md decision: warn-but-allow, no hard block).
