Add execution generation SQL to seed.sql using PostgreSQL's `generate_series` function. The executions simulate 60 days of Acme Corp automation usage with a gradual adoption curve.

**Growth curve pattern (60 days, ~500 total executions across all active automations):**

The demo org is a marketing agency that started using AIDEAS ~60 days ago. The growth should feel like real onboarding:

- **Week 1-2 (days 1-14):** Testing phase — ~3-4 runs/day total across all automations (~45 executions)
- **Week 3-4 (days 15-28):** Ramp-up — ~7-8 runs/day total (~105 executions)  
- **Week 5-6 (days 29-42):** Growing usage — ~10-12 runs/day total (~155 executions)
- **Week 7-8+ (days 43-60):** Full production — ~12-14 runs/day total (~195 executions)

Total: ~500 executions

**Implementation approach:**
For each active Acme automation (chatbot, content pipeline, lead nurture when it was active, executive report, social scheduler), generate a batch of executions using generate_series with different intervals per time period.

Example for the chatbot (heaviest usage automation):
```sql
-- Weeks 1-2: ~2 runs/day for chatbot
INSERT INTO public.automation_executions (...)
SELECT
    gen_random_uuid(),
    'au111111-0000-0000-0000-000000000001',
    CASE WHEN random() < 0.95 THEN 'success' ELSE 'error' END,
    NOW() - INTERVAL '60 days' + (n * INTERVAL '12 hours'),
    NOW() - INTERVAL '60 days' + (n * INTERVAL '12 hours') + (INTERVAL '1 second' * (random() * 10 + 3)::int),
    (random() * 8000 + 2000)::INTEGER,
    ('{"conversations": ' || (random()*15+3)::INTEGER || '}')::jsonb,
    CASE WHEN random() < 0.95 THEN ('{"handled": ' || (random()*12+2)::INTEGER || '}')::jsonb ELSE NULL END,
    CASE WHEN random() >= 0.95 THEN 'Webhook timeout after 30s' ELSE NULL END,
    'schedule'
FROM generate_series(1, 28) AS n;
```

Repeat with different parameters for each automation and each time period. Key rules:
- **95% success, 5% error** overall
- `duration_ms`: 2000-10000 for chatbot, 5000-15000 for reports, 3000-8000 for emails
- `input_data`: Relevant JSONB per automation type (conversations, emails_queued, invoices_received, etc.)
- `output_data`: NULL for errors, relevant completion data for success
- `error_message`: Realistic errors for failures (webhook timeout, API rate limit, authentication expired, connection refused, etc.) — vary the messages
- `triggered_by`: mostly 'schedule', some 'webhook' for chatbot, some 'manual'
- Timestamps: Use `NOW() - INTERVAL '60 days' + offset` pattern. Add small random variance to avoid perfectly uniform spacing.

**For the paused automation (lead nurture):** Generate executions only for the first ~40 days (it was paused 20 days ago). This explains why it's paused — it has history.

**For the in_setup automation (invoice processing):** NO executions — it hasn't been activated yet.

Generate at least 4-5 separate generate_series blocks to cover different automations and time periods. Target ~500 total rows.
