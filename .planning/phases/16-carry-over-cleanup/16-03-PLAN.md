From web/src/lib/actions/auth.ts (lines 23-42, EXISTING — do not modify):
```typescript
async function verifyRecaptcha(token: string): Promise<boolean> {
  const secret = process.env.RECAPTCHA_SECRET_KEY
  if (!secret) {
    console.warn('[reCAPTCHA] RECAPTCHA_SECRET_KEY not set — skipping verify')
    return true
  }
  // ... actual verify with Google ...
}
```

This means:
- If server's `RECAPTCHA_SECRET_KEY` is missing → server skips verification entirely (regardless of token).
- If server's `RECAPTCHA_SECRET_KEY` is present → server posts `secret + response=token` to Google. An empty `response` will be rejected by Google's siteverify endpoint (`success: false`), so the client bypass fails closed at the server.

This is the defense-in-depth design: client bypass leaking to prod accidentally would still be blocked at the server because production deploys MUST have `RECAPTCHA_SECRET_KEY` set.

From web/src/components/auth/signup-form.tsx (CURRENT lines 45-59 — to be edited):
```tsx
const handleFormSubmit = async (data: SignupFormData) => {
  // 1. Get reCAPTCHA token (bypass in development when not configured)
  let captchaToken = "dev-bypass";
  if (executeRecaptcha) {
    const token = await executeRecaptcha("signup");
    if (!token) {
      setError("root", { message: t("errors.captchaFailed") });
      return;
    }
    captchaToken = token;
  } else if (process.env.NEXT_PUBLIC_RECAPTCHA_SITE_KEY) {
    // reCAPTCHA configured but not loaded — real failure
    setError("root", { message: t("errors.captchaFailed") });
    return;
  }
  // ...
};
```

Current logic flaws (per audit):
1. Default token is `"dev-bypass"` (a non-empty literal). Server's `verifyRecaptcha("dev-bypass")` with a real `RECAPTCHA_SECRET_KEY` would POST `dev-bypass` to Google, which would reject it with `success: false`. So the bypass currently relies on the server ALSO having no secret — but the bypass triggers solely on `executeRecaptcha` being undefined, with no env-aware gate.
2. The `executeRecaptcha` hook returns undefined when the `<GoogleReCaptchaProvider>` isn't mounted. The provider is conditionally mounted somewhere (likely in a layout) based on the site key being present. If the key is missing in prod, `executeRecaptcha` is undefined AND the user doesn't see a captcha challenge — silent bypass to server which would reject the literal `"dev-bypass"` only because secret is set. This is fragile and depends on TWO env vars staying in sync.

Target logic:
- Compute `const isDevWithoutKey = process.env.NODE_ENV !== 'production' && !process.env.NEXT_PUBLIC_RECAPTCHA_SITE_KEY` ONCE at the top of the handler.
- If `isDevWithoutKey`: skip entirely, set `captchaToken = ""` (empty string per CONTEXT.md decision; matches server's `verifyRecaptcha("")` behavior when its own secret is missing), emit ONE `console.warn`, proceed.
- Else if `executeRecaptcha`: run it; on failure show error.
- Else (key configured but provider didn't load — prod with config issue, or dev with key but provider init bug): hard-fail with the existing `t("errors.captchaFailed")` error, same as today.

`process.env.NODE_ENV` is statically inlined by Next.js / Webpack at build time. In Vercel production builds it is always `"production"`. There is no runtime path where `NODE_ENV !== 'production'` evaluates true in a production bundle. This guarantee is what makes the dev-only bypass safe.

`process.env.NEXT_PUBLIC_RECAPTCHA_SITE_KEY` is also statically inlined for `NEXT_PUBLIC_*` vars in client bundles. Both checks happen at build time.

  let captchaToken = "";
