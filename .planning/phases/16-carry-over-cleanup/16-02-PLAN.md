From web/src/lib/auth/assert-org-membership.ts:
```typescript
type OrgRole = "owner" | "admin" | "operator" | "viewer";

export async function assertOrgMembership(
  supabase: SupabaseClient,
  orgId: string,
  allowedRoles?: OrgRole[]    // defaults to ["owner", "admin", "operator"]
): Promise<{ error: string } | null>;
```

Behavior:
- Calls `supabase.auth.getUser()` internally — caller does NOT need to pre-fetch the user before calling this helper. (But existing code in `saveCompanyName`/`saveHourlyCost` already does `getUser()` for other reasons; that call MAY remain and `assertOrgMembership` will simply re-check it. No harm, no extra round trips that matter.)
- Returns `{ error: "unauthorized" }` if no session.
- Returns `{ error: "forbidden" }` if user is not in `organization_members` for `orgId` with one of `allowedRoles` AND `is_active=true`.
- Returns `null` on success.
- Already logs `console.error` with structured context on rejection. Caller does NOT need to add additional logging.

Reference pattern (from `updateAutomationStatus` in `web/src/lib/actions/automations.ts`, the audit-cited canonical usage):
```typescript
const supabase = await createServerClient();
const denied = await assertOrgMembership(supabase, orgId, ["owner", "admin"]);
if (denied) return denied;
// ... proceed with admin client write ...
```

Current inline pattern in BOTH actions (to be replaced):
```typescript
const { data: member } = await supabase
  .from('organization_members')
  .select('role')
  .eq('user_id', user.id)
  .eq('organization_id', orgId)
  .single()

if (!member || !['owner', 'admin'].includes(member.role)) {
  return { error: 'unauthorized' }
}
```

Note the inline version returns `'unauthorized'` for BOTH "no session" and "wrong role". The helper distinguishes them: `'unauthorized'` for no session, `'forbidden'` for wrong role. This is a behavioral improvement consistent with `updateAutomationStatus`. UI surfaces (settings forms) already handle generic `error` strings, so the change of error code is a benign correctness fix, not a regression. Document this in the SUMMARY.

Inline check is also missing `is_active=true` filter; helper enforces it. This is a security improvement (a deactivated owner should not be able to update org settings) and is intentional.
