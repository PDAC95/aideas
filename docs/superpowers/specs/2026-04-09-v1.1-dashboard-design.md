# AIDEAS v1.1 — Core Dashboard Experience
*Design Spec · 2026-04-09*

---

## 1. Scope

Milestone v1.1 delivers the complete customer-facing dashboard with 5 sections + notifications + seed data. This is the first milestone where the user interacts with AIDEAS business features beyond authentication.

### Sections

| Section | Route | Purpose |
|---|---|---|
| Home | `/dashboard` | 10-second status: KPIs, automation list, activity feed |
| My Automations | `/dashboard/automations` | Inventory of user's automations with detail views |
| Catalog | `/dashboard/catalog` | Browse 66+ templates, view details, request with payment |
| Reports | `/dashboard/reports` | Impact metrics, weekly charts, per-automation breakdown |
| Billing | `/dashboard/billing` | Monthly charges, payment history, Stripe Customer Portal |
| Settings | `/dashboard/settings` | Profile, preferences (language, hourly cost), security |
| Notifications | (dropdown in sidebar) | Bell icon with unread count and notification list |

### Out of scope for v1.1

- Real-time chat with AIDEAS team (v1.2)
- Team management / invitations (v1.2)
- Admin panel for AIDEAS team (v1.2)
- Status update notes from AIDEAS team during setup (v1.2)
- Empty states for zero-automation users (v1.2 — seed data covers v1.1)

## 2. Architecture

### Data flow pattern: Hybrid C

```
READS  → Supabase direct from Next.js Server Components (RLS enforces org isolation)
WRITES → FastAPI endpoints (business logic, Stripe, notifications)
STRIPE → FastAPI exclusively (webhooks, checkout sessions, subscriptions, customer portal)
```

### Why Hybrid C

- RLS already enforces tenant isolation on all tables — no need to duplicate in API for reads
- Stripe requires server-side integration — FastAPI already has JWT validation and rate limiting
- Settings writes go direct to Supabase (no business logic, RLS permits own-profile updates)
- Mobile API (if ever needed) already has FastAPI covering all writes

### Route structure

```
web/src/app/(dashboard)/
  dashboard/
    page.tsx                    → Home
  automations/
    page.tsx                    → My Automations list
    [id]/page.tsx               → Automation detail
  catalog/
    page.tsx                    → Catalog grid
    [slug]/page.tsx             → Template detail + request flow
  reports/
    page.tsx                    → Reports
  billing/
    page.tsx                    → Billing
  settings/
    page.tsx                    → Settings
```

## 3. Schema changes

### Existing schema review

The v1.0 migration (`20260305000002_automation_business.sql`) already created these tables. The v1.1 migration must ALTER them, not recreate.

**Existing `automation_templates` columns:** id, name, slug, description, category (CHECK: customer_service, documents, marketing, sales, operations), icon, features[], use_cases[], config_schema, pricing_tier, is_active, is_featured, sort_order, created_at, updated_at.

**Existing `automations` columns:** id, organization_id, template_id, name, description, status (CHECK: draft, pending_review, active, paused, failed, archived), config, last_run_at, next_run_at, error_message, created_at, updated_at, deleted_at.

**Existing `automation_requests` columns:** id, organization_id, template_id, user_id, title, description, urgency, status (CHECK: pending, in_review, approved, completed, rejected), notes, completed_at, created_at, updated_at, deleted_at.

**Existing `subscriptions` columns:** id, organization_id, stripe_customer_id, stripe_subscription_id, plan, status, billing_cycle, current_period_start, current_period_end, cancel_at, cancelled_at, trial_start, trial_end, created_at, updated_at. Constraint: one subscription per org.

### Modified: `automation_templates`

Add new columns for catalog pricing and metrics:

| Column | Type | Purpose |
|---|---|---|
| `setup_price` | integer | Setup price in cents (Stripe convention) |
| `monthly_price` | integer | Monthly maintenance price in cents |
| `setup_time_days` | integer | Estimated setup time in business days |
| `industry_tags` | text[] | Array: retail, salud, legal, inmobiliaria, restaurantes, general |
| `connected_apps` | text[] | Array: WhatsApp, OpenAI, HubSpot, Gmail, etc. |
| `typical_impact_text` | text | Human-readable impact description for catalog detail |
| `avg_minutes_per_task` | integer | Minutes saved per execution (for hours-saved calculation) |
| `activity_metric_label` | text | Label for activity metric (e.g., "conversaciones resueltas", "leads calificados") |

Modify existing `category` CHECK constraint to expand values: customer_service, documents, marketing, sales, operations → **add**: productividad, reportes, agentes_ia. (ALTER column DROP old CHECK, ADD new CHECK.)

Existing columns `features[]` and `use_cases[]` remain — `features` will store catalog feature bullets, `use_cases` will store use case descriptions.

### Modified: `automations`

| Column | Type | Change |
|---|---|---|
| `stripe_subscription_id` | text | ADD — links to Stripe subscription for monthly billing |
| `status` CHECK | text | ALTER — add `in_setup` to allowed values: draft, pending_review, **in_setup**, active, paused, failed, archived |

### Modified: `automation_requests`

The existing table was designed for "request a custom automation" (title, description, urgency). For v1.1, we add catalog purchase fields alongside the existing columns:

| Column | Type | Change |
|---|---|---|
| `stripe_checkout_session_id` | text | ADD — Stripe Checkout session for setup payment |
| `request_type` | text | ADD — CHECK: 'custom', 'catalog_purchase'. Default 'custom' for backward compat |
| `status` CHECK | text | ALTER — expand: pending, in_review, approved, completed, rejected → **add**: pending_payment, paid, in_setup, cancelled |

This way the table supports both custom requests (v1.0 flow) and catalog purchases (v1.1 flow).

### Existing: `subscriptions` — model conflict resolution

The v1.0 `subscriptions` table is one-per-org (plan-based: starter/pro/business). The v1.1 billing model is per-automation (each automation has its own Stripe subscription).

**Decision: keep `subscriptions` as the org-level Stripe customer record, use `automations.stripe_subscription_id` for per-automation billing.**

- `subscriptions.stripe_customer_id` → the org's Stripe Customer (needed for Checkout, Portal)
- `subscriptions.plan`, `billing_cycle`, period fields → not used in v1.1 (plan-based pricing is out of scope), but kept for future use
- Each `automations` row with `stripe_subscription_id` → individual Stripe Subscription for monthly maintenance
- The `subscriptions` row is created when the org's first automation is purchased (to store `stripe_customer_id`)

### Existing: `organizations.settings` (JSONB)

New key used: `hourly_cost` (number) — user's configured cost per hour for value estimation in Reports.

## 4. FastAPI endpoints (new)

All new endpoints require JWT authentication via `Depends(get_current_user)`.

| Endpoint | Method | Purpose | Details |
|---|---|---|---|
| `POST /api/v1/automations/{template_id}/request` | POST | Request automation | Creates `automation_requests` record, creates Stripe Checkout Session with setup_price, returns `{ checkout_url }` |
| `POST /api/v1/automations/{id}/pause` | POST | Pause automation | Pauses Stripe subscription, updates automation status to `paused`, creates notification |
| `POST /api/v1/automations/{id}/resume` | POST | Resume automation | Resumes Stripe subscription, updates automation status to `active`, creates notification |
| `POST /api/v1/automations/{id}/cancel` | POST | Cancel automation | Cancels Stripe subscription, updates automation status to `archived`, creates notification |
| `POST /api/v1/billing/portal` | POST | Stripe Customer Portal | Creates portal session for org's stripe_customer_id, returns `{ portal_url }` |
| `GET /api/v1/billing/history` | GET | Payment history | Lists invoices + charges from Stripe API for the org |
| `POST /api/v1/stripe/webhooks` | POST | Stripe webhooks | Handles: `checkout.session.completed`, `invoice.paid`, `invoice.payment_failed`, `customer.subscription.updated`, `customer.subscription.deleted` |

### Stripe webhook flow

```
checkout.session.completed →
  1. Update automation_requests.status = 'paid' → 'in_setup'
  2. Create automations record (status: 'in_setup')
  3. Create notification: "Tu {name} está siendo configurada"

invoice.paid →
  1. Create notification: "Cobro mensual procesado: ${amount}"

invoice.payment_failed →
  1. Create notification (type: action_required): "Problema con tu pago"

customer.subscription.updated →
  1. Sync automation status with subscription status

customer.subscription.deleted →
  1. Update automation status to 'archived'
  2. Create notification: "{name} fue cancelada"
```

### Activation flow (manual for v1.1)

When the AIDEAS team finishes configuring an automation, they use a protected FastAPI endpoint:

| Endpoint | Method | Purpose |
|---|---|---|
| `POST /api/v1/admin/automations/{id}/activate` | POST | Activate automation after setup |

This endpoint requires service_role or a future admin JWT. It:
1. Creates Stripe Subscription (monthly_price from template, attached to org's stripe_customer_id from `subscriptions` table)
2. Updates `automations` record: status = 'active', stripe_subscription_id = new subscription ID
3. Updates `automation_requests` record: status = 'completed'
4. Creates notification: "Tu {name} está lista y activa"

In v1.2, this will be triggered from the admin panel UI. For v1.1, it's callable via curl/script with the service_role key.

## 5. Section: Home

### Layout

Top: greeting with user's first name.
Below: 3 summary cards (automations active, tasks completed this week, hours saved this month).
Below: compact list of user's automations with status and daily metric.
Below: activity feed (last 10-15 execution events).
CTA: "+ Nueva automatizacion" button linking to catalog.

### Components

| Component | Data source |
|---|---|
| `DashboardGreeting` | `profiles.first_name` via Supabase server client |
| `SummaryCards` | Aggregated queries on `automations` (count active) + `automation_executions` (count this week, sum hours this month using template's `avg_minutes_per_task`) |
| `AutomationsList` | `automations` JOIN `automation_templates` + today's execution count |
| `ActivityFeed` | `automation_executions` last 15, formatted by template type |
| `NewAutomationButton` | Static link to `/dashboard/catalog` |

### Metric calculations

- **Active automations**: `SELECT count(*) FROM automations WHERE org_id = $1 AND status = 'active'`
- **Tasks this week**: `SELECT count(*) FROM automation_executions ae JOIN automations a ON ae.automation_id = a.id WHERE a.organization_id = $1 AND ae.started_at >= now() - interval '7 days' AND ae.status = 'success'`
- **Hours saved this month**: Sum of (successful executions this month × template's `avg_minutes_per_task`) / 60

## 6. Section: My Automations

### List view (`/dashboard/automations`)

Filterable list of all user's automations. Filters: All, Active, In Setup, Paused (client-side filter).

Each card shows: name, category, connected apps, status badge (green/yellow/gray/red), monthly metric, monthly price.

### Detail view (`/dashboard/automations/[id]`)

- 3 KPI cards: metric count this month, hours saved, monthly charge
- Activity timeline: last 20 executions with formatted labels per type
- Weekly bar chart: executions per week (last 4 weeks)
- Action buttons: Pause/Resume (POST to FastAPI), link to Reports

### Automation states

| Status | Badge | Metric shown | Available actions |
|---|---|---|---|
| `active` | Green | Activity metric + price | Pause |
| `in_setup` | Yellow | "Solicitada el {date} - Setup en progreso" | None |
| `paused` | Gray | "Pausada desde {date}" | Resume, Cancel |
| `failed` | Red | Error message | None (contact support in v1.2) |

## 7. Section: Catalog

### Grid view (`/dashboard/catalog`)

Two filter layers:
- **Industry chips** (top): Todas, Retail, Salud, Legal, Inmobiliaria, Restaurantes
- **Category tabs**: Mas populares, Ventas, Marketing, Atencion al cliente, Documentos, Productividad, Reportes, Agentes IA

Grid of cards (3 cols desktop, 1 mobile). Each card: name, one-line description, connected app tags, setup price.

Data: `automation_templates WHERE is_active = true`, filtered client-side.

### Detail view (`/dashboard/catalog/[slug]`)

Full template info: description (business language), connected apps + prerequisites, typical impact text, setup time estimate, setup price + monthly price, "Solicitar esta automatizacion" button.

### Request flow

```
1. User clicks "Solicitar esta automatizacion"
2. Frontend POST /api/v1/automations/{template_id}/request
3. FastAPI:
   a. Creates automation_requests (status: pending_payment)
   b. Creates Stripe Checkout Session (setup_price, success_url, cancel_url)
   c. Returns { checkout_url }
4. Frontend redirects to Stripe Checkout
5. User pays
6. Stripe sends checkout.session.completed webhook
7. FastAPI webhook handler:
   a. Updates automation_requests (status: in_setup)
   b. Creates automations record (status: in_setup)
   c. Creates notification
8. User returns to dashboard, sees automation in "En setup" state
```

## 8. Section: Reports

### Layout

Top: period selector dropdown (This month, Last month, Last 3 months).
Below: 3 impact KPI cards (tasks completed, hours saved, estimated value).
Below: weekly activity bar chart (last 60 days, ~8 bars).
Below: breakdown table (per automation: name, metric label + count, hours saved).

### Estimated value (optional layer 3)

If `organizations.settings.hourly_cost` is set:
- Show card: value = hours_saved * hourly_cost, with note "Basado en tu costo/hora: ${hourly_cost}/h"

If not set:
- Hide value card, show link: "Configura tu costo/hora para ver el valor estimado" → `/dashboard/settings`

### Components

| Component | Data source |
|---|---|
| `PeriodSelector` | Client-side state, changes query range |
| `ImpactSummaryCards` | Aggregated `automation_executions` for period + `avg_minutes_per_task` from templates |
| `WeeklyActivityChart` | `automation_executions` grouped by week |
| `AutomationBreakdownTable` | `automation_executions` grouped by `automation_id`, JOIN template for metric label |

## 9. Section: Billing

### Layout

Top: summary card (total monthly active, next charge date, manage payment button).
Below: monthly charges table (per active automation + "en setup" with asterisk).
Below: payment history table (from Stripe API).
Bottom: link to full invoices in Stripe.

### Data flow

- **Monthly charges**: `automations` JOIN `automation_templates` for `monthly_price`, local data
- **Payment history**: FastAPI `GET /api/v1/billing/history` → Stripe API (invoices + charges)
- **Manage payment**: FastAPI `POST /api/v1/billing/portal` → Stripe Customer Portal URL → redirect
- **Next charge date**: From Stripe subscription's `current_period_end`

### Design decisions

- Billing is read-only. Pause/cancel actions live in My Automations detail view.
- Payment history comes from Stripe as source of truth — no duplication in our database.
- Stripe Customer Portal handles method of payment changes, invoice downloads, etc.

## 10. Section: Settings

### Layout

Three blocks: Profile, Preferences, Security.

### Profile block

- Avatar upload (Supabase Storage, bucket: `avatars`)
- Name (editable)
- Email (read-only, from Supabase Auth)
- Company name (editable, maps to `organizations.name`)

### Preferences block

- Language selector: Espanol / English (sets `NEXT_LOCALE` cookie)
- Hourly cost input: number field, saved to `organizations.settings.hourly_cost`

### Security block

- Change password: `supabase.auth.updateUser({ password })`
- Active sessions count + "Close all other sessions": `supabase.auth.signOut({ scope: 'others' })`

### All writes go direct to Supabase

No FastAPI needed — these are self-profile updates protected by RLS. No business logic side effects.

## 11. Notifications

### UI

Bell icon in sidebar header with unread count badge. Click opens dropdown with last 20 notifications, each showing icon (by type), title, message, relative timestamp. "Mark all as read" button at bottom.

### Notification types

| Event | Type | Created by |
|---|---|---|
| Setup payment confirmed | `success` | Stripe webhook handler |
| Automation in setup | `info` | Stripe webhook handler |
| Automation activated | `success` | Admin/script |
| Monthly payment processed | `info` | Stripe webhook handler |
| Payment failed | `action_required` | Stripe webhook handler |
| Automation paused | `warning` | FastAPI pause endpoint |
| Automation cancelled | `info` | FastAPI cancel endpoint |

### Data

- Read: Supabase direct (notifications table, RLS filtered by user_id)
- Mark as read: Supabase direct UPDATE
- Create: FastAPI (as side effect of business operations)
